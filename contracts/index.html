Executive Summary

MNX is closest to a transparent, per-market isolated-margin exchange contract design. The core risk and settlement rules live in Solidity contracts: Perpetual, IsolatedTrader, FundingOracle, BackendPriceOracle, IsolatedADL, LiquidityVault, and MarginBank. The high-throughput runtime is still important, but it submits calls into these contracts rather than replacing the contract rules.

MNX

Per-market isolated margin. On-chain uniform-batch trade settlement. Permissioned settlement operators, but contract-enforced margin, funding settlement, liquidation bypass, backstop inheritance, ADL eligibility, and oracle freshness checks.

Latest public IDEX-lineage perps contracts are the Kuma/Ikon contracts. They are fully cross-margined, use a dispatcher-only settlement model, hold funds in a separate Custodian, and validate funding, liquidation, and ADL paths in Solidity libraries.

High-Signal Differences

  • Funding: MNX computes a one-hour funding index from trade-vs-oracle premium samples in FundingOracle, or accepts an off-chain override when enabled. IDEX/Kuma publishes an off-chain funding multiplier every eight hours and lazily applies it to wallets. Hyperliquid pays hourly using a centralized-exchange-style formula sampled from impact prices and validator oracle prices. StarkEx supports funding ticks through global funding indices, but the exchange application supplies those ticks.
  • Liquidations: MNX has signed forced-liquidation order support plus a backstop inheritance path into a liquidity vault child account. IDEX/Kuma liquidates wallets or positions into the insurance fund or exit fund, with contract-side bankruptcy and margin checks. Hyperliquid first routes liquidation flow to the book, then uses the liquidator vault below a deeper threshold. StarkEx liquidations are batch transactions that improve an under-collateralized position and are validity-proven.
  • Automatic deleveraging (ADL): MNX ADL is a dedicated trader contract used when funding-adjusted equity is non-positive; it closes against profitable opposite-side accounts at the bankrupt account's bankruptcy price. IDEX/Kuma has multiple deleverage functions for in-maintenance acquisition, insurance-fund closure, exit acquisition, and exit-fund closure. Hyperliquid ranks profitable opposite-side users by unrealized profit and leverage. StarkEx calls this the deleverage flow and allows it when the target position value is negative.
  • Oracles: MNX uses a Chainlink-compatible BackendPriceOracle with permissioned updaters, per-update basis-point limits, one update per block, and freshness checks in Perpetual. IDEX/Kuma lazily publishes signed index prices through whitelisted adapters and maintains a separate oracle-price path for on-chain views. Hyperliquid validators publish spot oracle prices roughly every three seconds, and mark price combines oracle, Hyperliquid book, and external perp markets. StarkEx uses signed oracle price ticks with configured keys and quorum.

Glossary

Funding

Periodic payments between longs and shorts designed to pull the perpetual contract price toward the reference spot or index price. Positive funding usually means longs pay shorts.

Liquidation

Forced reduction or transfer of a risky position after account equity falls below the maintenance requirement. A venue may route this through a book, an insurance fund, a liquidator vault, or a deleveraging path.

Automatic deleveraging

ADL is the final risk process where profitable opposite-side traders are forced to reduce exposure against an underwater account or fund position. It prevents bad debt from remaining in the system.

Oracle and mark price

An oracle price is the external reference price. A mark price is a fair-value price used for margin and liquidation; it may combine oracle and order-book data.

Price Semantics

There is no universal "true price" across these systems. Each system has a risk/reference price used for solvency checks, and several systems use a different settlement or execution price once a liquidation or ADL path is allowed. This section ignores off-chain price construction and focuses on what the smart contract, proof system, or native state transition reads after a price has been accepted into protocol state.

MNX

Inside MNX contracts, the closest thing to "true" or "fair" price is the oracle price read by Perpetual._loadContext() and Perpetual._readOraclePrice(). The contract requires a positive supported price and checks freshness against either current block time or the cached global-index timestamp, depending on the path.

  • Margin and liquidation checks: use oracle price. IsolatedTrader._verifyForcedLiquidationOrder computes the account margin ratio with oracle price and requires it to be below maintenance.
  • Funding: has two prices. The premium sample is trade price versus oracle price. The funding index accrues with oracle price once the rate is known.
  • ADL: uses oracle price to validate eligibility and min/max price checks, but actual ADL settlement returns bankruptcy price from IsolatedADL.
  • Regular trades: settle at the submitted fill/uniform batch price; that is not treated as the universal fair price for solvency checks.

Key local evidence: Perpetual.sol, IsolatedTrader.sol, IsolatedADL.sol.

StarkEx Perpetual

StarkEx is not an ordinary Solidity perp contract. The relevant contract-level semantics are the validity-proven state transitions. The reference price is the internal current price from a valid oracle price tick. The application submits signed oracle updates; StarkEx verifies the configured signer/quorum/median/timing constraints and then uses the accepted tick in later validity checks.

  • Liquidation: StarkEx checks that total account value divided by total maintenance margin requirement is below 1 and improves after liquidation.
  • ADL/deleverage: the target position must be negative according to the most current prices, and StarkEx validates opposite signs, reduced exposure, and value/risk constraints.
  • Funding: funding ticks update global funding indices; the collateral change is derived from the new index minus the position's cached index.

Primary references: Oracle price tick, Liquidation flow, Deleverage flow, Funding tick.

IDEX/Kuma Ikon

For normal Ikon risk checks, the contract price is market.lastIndexPrice. Index price is distinct from oracle-price accessors. The core margin and liquidation libraries use index price for account value and margin requirements, while OraclePriceMargin is a separate oracle-price path for exits and on-chain views.

  • Margin: IndexPriceMargin values positions with market.lastIndexPrice and computes initial and maintenance requirements from that same index price.
  • Funding: Funding.publishFundingMultiplier_delegatecall stores a multiplier derived from fundingRate and market.lastIndexPrice.
  • Liquidation: wallet-in-maintenance liquidation validates bankruptcy quote quantities using index-price account value; deactivated-market liquidation uses the stored indexPriceAtDeactivation; wallet-exit liquidation may use exit price or bankruptcy price.
  • ADL: uses path-specific settlement prices: bankruptcy price for in-maintenance acquisition, insurance-fund entry/cost basis for insurance closure, exit price or bankruptcy price for exit acquisition, and index price or exit-fund bankruptcy price for exit-fund closure.

Primary references: IndexPriceMargin.sol, Funding.sol, PositionInDeactivatedMarketLiquidation.sol.

Hyperliquid

Hyperliquid's perp checks live in HyperCore, not in public Solidity contracts. The protocol has two major accepted price concepts: validator-published spot oracle price and mark price. Oracle price feeds funding and is one input to mark price. Mark price is the native risk price.

  • Funding: uses spot oracle price for notional and compares impact bid/ask prices against oracle price to compute the premium component.
  • Liquidation: uses mark price for margining and liquidation risk.
  • ADL: triggers when account value or isolated position value becomes negative; profitable opposite-side accounts are ranked by unrealized PnL and leverage, and closure occurs at previous mark price.
  • Mark construction: mark price is a median of oracle plus EMA basis, Hyperliquid book signals, and external perp-market prices.

Primary references: Oracle, Robust price indices, Funding, Liquidations, Auto-deleveraging.

MNX Contract-Level Exposition

MNX's contract design is per-market and isolated-margin. User collateral is held by MarginBank; market positions and funding indices are held by Perpetual; trade semantics are delegated to trader contracts such as IsolatedTrader for signed orders and forced liquidations, and IsolatedADL for bankrupt-account deleveraging. The backend runtime controls batching, order matching, keepers, liquidation planning, and oracle submissions, but the material risk checks are repeated or enforced in contracts.

Core Contract Roles

  • MarginBank: holds ERC-20 collateral, tracks bank balances, enforces total collateral coverage, and lets whitelisted market operators move margin between accounts.
  • Perpetual: owns market position state, margin-ratio checks, global funding index updates, trade settlement, funding settlement, backstop inheritance, ADL eligibility, and oracle freshness checks.
  • FundingOracle: receives trade premium samples from Perpetual, computes or stores the funding rate, and exposes a per-second rate to the market.
  • BackendPriceOracle: Chainlink-compatible backend oracle with permissioned updaters, optional max basis-point move, and one update per block.
  • IsolatedTrader: validates maker/taker signatures, order bounds, market-take bounds, margin impact, and forced-liquidation orders.
  • IsolatedADL: validates bankrupt maker and profitable opposite-side taker, prices ADL at bankruptcy, and returns zero-fee trade results to Perpetual.
  • LiquidityVault: implements the backstop vault/child-vault mechanism used when a low-equity but positive-equity position can be inherited.

MNX Funding

The funding implementation has two separate layers: rate calculation in FundingOracle, and account settlement in Perpetual.

Rate calculation

  • Normal FBA trade batches call FundingOracle.recordTrade(tradePrice, oraclePrice, timestamp) through Perpetual.trade. The recorded ratio is (trade price - oracle price) / oracle price, accumulated as a time-weighted premium sample.
  • FundingOracle.setFundingRate() can be triggered through Perpetual.setFundingRate(). The oracle computes a one-hour average premium, applies a dampening factor by dividing by 24, clamps the result to the configured max hourly funding, then stores the per-second funding rate.
  • setOffChainFundingRate(int256) exists for the off-chain funding mode. It is restricted by offchainFROperator at the Perpetual level and by max-funding checks in FundingOracle.
  • The Guardian can disable or re-enable funding. When disabling, it attempts to refresh the global index first; when enabling, it refreshes before allowing future accrual.

Global index and settlement

  • Perpetual._loadContext(true) reads the oracle and funding rate, checks staleness against current time, and updates globalIndex.value using fundingRate * elapsedTime * oraclePrice. It emits GlobalIndexUpdate.
  • Perpetual._loadContext(false), used on the hot trade path, does not update the index but still requires the oracle update to be fresh enough relative to the cached global-index timestamp.
  • Before trades, Perpetual.trade applies pending funding for all touched accounts. There is also a permissionless settleFundingForAccounts(address[]) path used by the keeper to settle open positions in batches.
  • Funding settlement is asymmetric by side: longs use oldIndex - newIndex; shorts use newIndex - oldIndex. Positive settlement credits margin. Negative settlement debits margin with round-up; if margin is insufficient, the balance is floored at zero and the unpaid amount is emitted as forgiven funding.

Key local evidence: Perpetual.sol funding methods and _settlePendingFunding; FundingOracle.sol; runtime-funding-settlement.ts.

MNX Liquidations and Backstop Inheritance

MNX has two contract-level liquidation-style paths: order-book forced liquidation via IsolatedTrader, and backstop inheritance via LiquidityVault.

Forced liquidation orders

  • IsolatedTrader._verifyForcedLiquidationOrder requires a reduce-only market order, an existing position, a non-flipping reduction, and account margin ratio below maintenance when valued at the oracle price.
  • Perpetual.trade recognizes liquidation trader flags and relaxes normal account checks only for valid liquidation bypass scenarios. It still verifies margin-ratio transitions after all trades in the batch.
  • A liquidation cannot arbitrarily increase risk: when a position is under initial margin but above maintenance, the new margin ratio must improve; below maintenance, only reducing trades are allowed unless the trade is a valid liquidation or ADL path.

Backstop inheritance

  • attemptBackstopInheritance(address) is settlement-operator-only. It loads fresh context, funding-adjusts the account in memory, and requires positive equity below the configured backstop trigger fraction.
  • If the vault accepts the account, LiquidityVault.inheritPosition returns the target child vault, and Perpetual.completeBackstopInheritance transfers the full position and isolated margin to that child account.
  • The current high-throughput liquidation runtime submits attemptBackstopInheritance payloads for prepared backstop cases. That makes the current runtime behavior closer to Hyperliquid's liquidator-vault takeover than to a pure order-book liquidation bot, even though forced-book liquidation logic exists in the contracts.

Key local evidence: IsolatedTrader.sol, Perpetual.sol, LiquidityVault.sol, runtime-liquidations.ts.

MNX Automatic Deleveraging

MNX ADL is explicit and contract-routed. The runtime plans ADL legs, but settlement goes through Perpetual.trade(accounts, trades) using IsolatedADL as the trader contract.

  • Perpetual.isAccountAdlEligible excludes flat positions and special binary 1x floor cases, then requires funding-adjusted equity to be less than or equal to zero.
  • IsolatedADL.trade treats the maker as the bankrupt account and the taker as the surplus opposite-side account. It rounds the oracle price to tick size for validation but returns a settlement price equal to the bankrupt account's bankruptcy price.
  • Bankruptcy price is derived from settled balance: for a long, (open interest - margin) / quantity; for a short, (open interest + margin) / quantity.
  • The ADL contract requires maker and taker to be opposite sides, maker margin ratio <= 0, taker margin ratio > 0, no increase of the maker position, and no negative taker equity after applying the ADL leg.
  • The runtime planner ranks profitable counterparties by a Hyperliquid-style score based on profit percentage and leverage used, then submits legs until the bankrupt exposure is closed or capacity is exhausted.

Key local evidence: IsolatedADL.sol, waterfall-adl-planner.ts, adl-submission.ts, runtime-adl.ts.

MNX Oracle Updates

  • BackendPriceOracle implements the Chainlink latestRoundData() shape. The owner configures an updater, optional max staleness, and optional max basis-point update size.
  • setPrice is updater-only, rejects non-positive or overflowing prices, enforces max move if configured, and writes at most once per block.
  • Perpetual._readOraclePrice requires the returned price to be supported, positive, and fresh. If oracleMaxStalenessSeconds is non-zero, updatedAt + maxStaleness must be at least the reference timestamp.
  • The runtime price updater aggregates external prices, removes outliers when enough sources exist, computes internal mark-price data, then submits bounded BackendPriceOracle.setPrice calls through the HTV1 relayer.
  • The oracle update guard rejects duplicate pending updates, prevents same-block repeats, and can bound a submitted target price by the oracle's max-update basis-point limit.

Key local evidence: BackendPriceOracle.sol, Perpetual.sol, price-updater.ts, oracle-price-update-guard.ts.

StarkEx Perpetual

StarkEx is best compared to MNX as a settlement and proof framework, not as a single public Solidity perp market. Its L1 contracts verify state updates and implement forced-operation escape machinery; the perps business logic is executed and proven in StarkEx batches.

Funding

StarkEx documents funding as a funding tick: a periodic global funding index update supplied by the operator/application. Funding affects a position only when a transaction touches that position, and multiple missed ticks are cumulatively applied. The collateral update is driven by the delta between the new global funding index and the position's cached index.

Relative to MNX, the closest concept is MNX's globalIndex and lazy account settlement. The difference is that MNX's funding-rate calculation is implemented in the local Solidity FundingOracle, while StarkEx describes funding ticks as application business logic sent into the StarkEx gateway and validity-proven.

Liquidations

StarkEx enforces maintenance margin, while the application is responsible for initial margin before submitting transactions. A liquidation can be submitted without the liquidated user's signature when the target position is below maintenance margin. StarkEx checks that the liquidated position is actually liquidatable, that the value-to-maintenance ratio improves, and that the synthetic holdings are reduced without changing sign.

Compared with MNX, StarkEx's liquidation is more like a validity-proven transaction type in a batch than a direct Solidity function such as attemptBackstopInheritance or Perpetual.trade.

Automatic Deleveraging

StarkEx calls this the deleverage flow. It applies when a position's total value becomes negative. The application matches that position with an opposite-side position, and StarkEx checks opposite signs, reduced total exposure, no sign flips, the deleverager's margin condition, valid asset IDs, valid position ownership, valid signatures where required, and a fairness condition preserving the value-to-risk ratio.

MNX and StarkEx are conceptually close here: both reserve ADL for accounts that have crossed into negative value. MNX exposes a concrete IsolatedADL Solidity trader contract; StarkEx exposes a proven transaction type.

Oracle Updates

StarkEx uses external oracle price feeds. The application collects signed price updates from registered oracle providers, aggregates them into an oracle price tick request, and sends the request to StarkEx. StarkEx validates configured signer keys, quorum, median price, and time-window constraints, then includes the price tick in a batch.

Compared with MNX's permissioned BackendPriceOracle.setPrice, StarkEx has more explicit signed-quorum semantics in the proof system. The tradeoff is that a reader must understand the StarkEx batch and application configuration rather than inspect a single oracle updater contract.

Important distinction: StarkEx gives users an L1 forced-operation and freeze/escape path. Users can submit forced withdrawals or forced trades on-chain; if the operator does not serve them within the grace period, the contract can be frozen and users can escape using the last oracle price tick. MNX has ordinary on-chain custody in MarginBank and delayed/signature withdrawal mechanics, but not the same StarkEx proof-system escape model.

Latest IDEX/Kuma Ikon Contracts

The latest public perps contracts in the IDEX lineage are the Kuma/Ikon contracts in kumabid/kuma-contracts-ikon. The README describes Ikon as a fully cross-margined USDC perpetual futures product with index pricing, funding payments, liquidation, ADL, and smart-contract custody. The Solidity code centers on Exchange_v1, with most logic split into delegatecall libraries.

Contract Architecture

  • Exchange_v1 tracks balances, markets, index prices, funding multipliers, exits, and dispatcher/admin/governance roles.
  • Custodian holds funds, while Exchange_v1 tracks user asset balances and settlement state.
  • executeTrade, withdraw, liquidations, ADL, publishIndexPrices, and publishFundingMultiplier are dispatcher-driven entry points, with contract validations in libraries.
  • Margin is cross-market by default, unlike MNX's isolated per-market accounting.

Funding

Ikon funding is published as a compact on-chain multiplier ledger. The dispatcher calls Exchange.publishFundingMultiplier(baseAssetSymbol, fundingRate). The Funding library requires the market to be active and the absolute funding rate to be lower than that market's maintenance margin fraction.

The stored value is funding rate * index price, with sign flipped so it can be directly multiplied by the wallet's base position to compute quote credit or debit. Funding periods are eight hours, aligned to 00:00, 08:00, and 16:00 UTC; missing periods are backfilled with zero multipliers. Funding is lazily applied on balance-changing actions, and applyOutstandingWalletFundingForMarket can proactively walk historical multipliers for inactive wallets.

Relative to MNX, Ikon moves rate calculation off-chain and commits only the multiplier. MNX can calculate funding rate on-chain from trade samples or accept an off-chain override, then updates a global index continuously by elapsed seconds.

Liquidations

Ikon has several liquidation paths exposed on Exchange_v1:

  • liquidatePositionBelowMinimum: transfers a dust-sized position to the insurance fund at current index price, with tolerance for small-position rounding.
  • liquidatePositionInDeactivatedMarket: closes a position in a deactivated market at the deactivation index price.
  • liquidateWalletInMaintenance: transfers all positions of a wallet below maintenance to the insurance fund at each position's bankruptcy price.
  • liquidateWalletInMaintenanceDuringSystemRecovery: similar, but routes to the exit fund during offline recovery.
  • liquidateWalletExit: handles an exited wallet at exit price or bankruptcy price.

The wallet-in-maintenance library applies outstanding funding to the liquidating and counterparty wallets before checks. It validates the liquidating wallet is below maintenance using index-price account value and maintenance requirement, calculates bankruptcy quote quantities, updates all positions, sweeps residual quote balance, and validates that the insurance fund remains above initial margin after normal liquidation.

Automatic Deleveraging

Ikon ADL is dispatcher-submitted but contract-validated. The main entry points are:

  • deleverageInMaintenanceAcquisition: reduces a wallet below maintenance directly against a counterparty at bankruptcy price.
  • deleverageInsuranceFundClosure: reduces an insurance-fund position against a counterparty at the insurance fund's entry or cost-basis price.
  • deleverageExitAcquisition: reduces an exited wallet against a counterparty at exit price or bankruptcy price, depending on exit account value.
  • deleverageExitFundClosure: reduces the exit fund at index price if the exit fund account value is positive, or at bankruptcy price if negative.

For in-maintenance acquisition, the ADL library applies funding to both wallets, confirms the liquidating wallet is below maintenance, validates the quote quantity at bankruptcy price, updates positions, then validates that the counterparty still meets maintenance margin. Counterparty ranking is an off-chain dispatcher concern; unlike Hyperliquid and MNX's runtime planner, the inspected Solidity entry points validate a chosen counterparty rather than defining a public ranking queue.

Oracle and Index Price Updates

Ikon distinguishes index pricing and oracle pricing. Index prices determine margin requirements, liquidation conditions, ADL conditions, withdrawals, transfers, and funding inputs. The dispatcher lazily publishes index prices immediately before dependent settlement operations through publishIndexPrices. The contract checks that each adapter is whitelisted, validates the payload through the adapter, rejects timestamps more than one day in the future, requires active markets, and requires strictly increasing timestamps.

Oracle-pricing accessors exist for on-chain analysis and wallet exits, but the README warns that oracle-price and index-price accessors can differ because the methodologies are different. This differs from MNX, where the per-market BackendPriceOracle is the direct Chainlink-shaped price feed read by Perpetual.

Hyperliquid

Hyperliquid's perps engine is not a normal Solidity contract system. The official docs describe the perps clearinghouse as a component of HyperCore execution state that manages margin balances and positions. HyperEVM is secured by the same consensus and can interact with HyperCore through read precompiles and a CoreWriter system contract, but the native perp funding, liquidation, ADL, and oracle mechanisms live in HyperCore.

Funding

Hyperliquid pays funding every hour. The documented rate follows a centralized-exchange-style formula for an eight-hour funding rate, then pays one eighth of that rate each hour. The formula is:

Funding Rate = Average Premium Index + clamp(interest rate - Premium Index, -0.0005, 0.0005)

The interest component is fixed at 0.01% per eight hours. The premium is sampled every five seconds and averaged over the hour. The funding payment is position_size * oracle_price * funding_rate; the spot oracle price, not the mark price, is used to convert size to notional. Hyperliquid caps funding at 4% per hour.

Compared with MNX, Hyperliquid's funding calculation is native clearinghouse logic. MNX's funding rate and account settlement are visible in Solidity, while Hyperliquid exposes current and historical funding through API/HyperCore state rather than a public EVM funding contract.

Liquidations

Hyperliquid liquidates when account equity falls below maintenance margin. Maintenance margin is half of the initial margin at max leverage. The engine first attempts to close positions by sending market orders to the book. If book liquidation fully or partially restores maintenance compliance, remaining collateral stays with the trader.

If account equity falls below two thirds of maintenance margin without successful book liquidation, a backstop liquidation occurs through the liquidator vault, a component strategy of HLP. Cross backstop liquidation transfers cross positions and cross margin; isolated backstop liquidation transfers only that isolated position and isolated margin. Liquidations use mark price, not a single book tick.

Compared with MNX, Hyperliquid's liquidator vault is conceptually close to MNX's LiquidityVault backstop inheritance. The implementation boundary differs: MNX executes the inheritance through Solidity calls; Hyperliquid performs it in native HyperCore clearinghouse logic.

Automatic Deleveraging

Hyperliquid ADL triggers when a user's account value or isolated position value becomes negative. Opposite-side users are ranked by unrealized profit and leverage used. The documented sorting index is (mark_price / entry_price) * (notional_position / account_value). Selected positions are closed at the previous mark price against the underwater user, preventing bad debt.

MNX deliberately mirrors this style in its runtime planner: profitable opposite-side accounts are ranked by a Hyperliquid-convention score, then settled through the IsolatedADL contract at bankruptcy price.

Oracle and Mark Price Updates

Hyperliquid validators publish spot oracle prices for each perp asset about every three seconds. Each validator computes a weighted median of major exchange spot mid prices; the final clearinghouse oracle is the weighted median of validator submissions, weighted by stake. Oracle prices feed funding and are an input to mark price.

The mark price is the median of three families of prices: oracle price plus a 150-second exponential moving average of Hyperliquid mid minus oracle, the median of Hyperliquid best bid/best ask/last trade, and a weighted median of external perp mid prices. Mark and oracle prices update approximately every three seconds and mark is used for margining, liquidations, trigger orders, and unrealized PnL.

Takeaways

1. MNX is more contract-explicit than Hyperliquid and more venue-specific than StarkEx.

MNX exposes the core risk mechanics in Solidity contracts. Hyperliquid exposes similar financial semantics through HyperCore state and APIs, while StarkEx exposes a proof framework and transaction types. For an auditor, MNX gives a narrower set of Solidity files to inspect; for a systems reviewer, Hyperliquid and StarkEx require understanding native/proof execution outside the EVM.

2. MNX and IDEX/Kuma differ most sharply on margin scope.

MNX is isolated by market. Ikon is cross-margined across markets. This changes every downstream risk process: funding debits in MNX hit isolated margin and can be forgiven if the bucket is empty; Ikon funding updates quote balance across a cross-margin wallet; Hyperliquid supports both cross and isolated; StarkEx's position model can include collateral and multiple synthetic assets.

3. MNX ADL is intentionally close to Hyperliquid's queue idea, but settlement price differs.

MNX's runtime ranks profitable opposite-side accounts using a Hyperliquid-style profit/leverage score. The MNX contract then settles at bankruptcy price through IsolatedADL. Hyperliquid docs describe closure at previous mark price. Ikon uses path-specific prices. StarkEx enforces a fairness condition rather than publishing a queue formula in the generic docs.

4. Oracle design is the largest trust-boundary difference.

MNX's oracle is a permissioned backend feed with contract-level movement and freshness limits. Ikon uses whitelisted adapters and lazy index publication. StarkEx uses signed oracle ticks with quorum in the validity system. Hyperliquid uses validator-published spot oracles and a robust mark-price construction. These are not interchangeable security models, even when they feed similar margin formulas.

Caveat: This is a contract-level research artifact, not a formal audit. It intentionally focuses on funding, liquidations, ADL, and oracle updates. It does not exhaustively cover deposits, withdrawals, governance upgrade risk, sequencer/liveness risk, bridge risk, API correctness, or market-maker strategy risk.
