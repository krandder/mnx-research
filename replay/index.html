Finding map

Original blocker

Policy mismatch in the source data

The initial 1M plan mixed spot, synthetic, hyperliquid-index, and perpetual records into an AAPL execution target. It also included reduce-only rows and orders below min size or funding envelope.

Finding: the replay needs an explicit executable policy before scale claims are meaningful.

Fixed for M03

Clean executable policy

M03 now uses perpetual-only rows, skips reduce-only orders for initially flat users, rejects below-min orders, and bounds initial margin for the baseline-executable target.

Finding: the data stream can produce a deterministic clean 1M baseline-executable plan.

Scale risk

Funding and user distribution

Smaller funding profiles failed before 10k orders. The first viable scale profile used 1,024 users and large per-user USDM funding.

Finding: a replay can fail from cumulative margin pressure before it reaches the system path we meant to test.

Guarded

Oracle staleness and P48

The global-index refresh failures were traced to local replay oracle freshness. Batch-count refresh was insufficient once batches slowed, so the harness added elapsed-time oracle refresh.

Finding: time matters in this exchange. Batch count is not a substitute for wall-clock freshness.

Regression added

Stale expiration cancellation

At scale, an order could expire and be removed milliseconds before another path still tried to cancel it. Exact-hash stale failures are now tolerated in the expiration loop.

Finding: lifecycle operations must be idempotent around already finalized orders.

Regression added

Stale matched-order cancellation

A later retry reached 70k orders and failed at batch 703 with No order found after the same order had expired. The matched-cancel finalizer now tolerates exact-hash stale cancellation failures.

Finding: expiration and post-match cleanup can race on the same order hash.

Open

Receipt reconciliation quarantine

The active retry has one quarantined mined receipt: an observed chain batch could not be reconciled with an expected sealed batch or prepared liquidation.

Finding: this is not yet fatal, but it is exactly the kind of chain/effect consistency issue that needs a dedicated research pass.

Open

Materializer and mirror lag

The active retry is producing many slow materializer batches and growing journal mirror lag while the critical settlement path remains aligned.

Finding: status and persistence subsystems may become the next scale gate before exchange execution fully fails.

Not covered

Liquidation and ADL in baseline

The no-market-maker baseline has produced no liquidations or ADL. That is expected for this replay shape and does not validate the liquidation or ADL systems.

Finding: liquidation/ADL coverage requires transformed price paths, lower collateral buffers, or market-maker-specific stress.

Failure timeline

M03 initial 1M planning

The replay was not yet an exchange test.

It was first a policy problem: too many source kinds and order shapes were being treated as if they were executable AAPL perpetual orders.

10k scale probes

Insufficient margin became the first practical gate.

Profiles with too few users or too little funding failed before reaching the exchange behavior under investigation. The viable shape moved to 1,024 users and larger per-user funding.

Post-test diagnostics

External snapshots misled the settlement analysis.

The first 10k diagnostics showed a position-applied versus settlement-applied gap. In-test diagnostics later showed the gap was a capture-timing artifact after teardown resumed background jobs.

100k attempt, batch 115

Global-index refresh hit stale oracle state.

AAPL crossed the local oracle max-staleness window. This produced refresh timeout behavior and later P48 signatures.

100k retry, batch 149

Batch-count oracle refresh was not enough.

The batch cadence crossed the old failure point but later slow batches made the wall-clock freshness window expire. The guard became time-aware.

100k retry, batch 697

Expiration and wakeup raced.

The runtime expired an order, then wakeup attempted to use that same order as live eight milliseconds later. Exact-hash stale expiration candidates are now non-fatal.

100k retry, batch 703

Matched-order cleanup had the sibling race.

A matched/post-match cancellation path hit No order found for an order that had already expired. Exact-hash stale matched cancellations are now non-fatal.

Active retry, batch 300

Lifecycle fixes are holding so far.

The current run has crossed 30k placed orders with zero API failures, zero reverts, and no recurrence of the stale-order crashes. The current watch item is performance and receipt reconciliation, not placement failure.

ADL stress research findings

Invariants that held

  • Child-vault bankrupt maker state stayed classified as child_vault through runtime ADL and relayer handoff.
  • Child-vault taker routing preserved taker_child_vault_id and did not require a taker user id.
  • Receipt routing applied child-vault taker and maker effects through child-vault state where expected.

Bugs found and fixed

  • ADL funds-flow balance deltas were previously batched only for user accounts, missing child-vault MarginBank balance effects.
  • ADL position margin-delta balance effects were previously published only for users, skipping child-vault margin changes.
  • Duplicate decoded position or settlement log indexes now fail before receipt effects are journaled.

Why ADL remains a separate workstream

The baseline replay is not naturally producing liquidations or ADL. ADL correctness is therefore being tested with targeted stress scenarios. The right next bridge is to make replay scenarios that deliberately enter liquidation and ADL states, then compare runtime and receipt behavior to the targeted ADL invariants.

Next decisions

Decision Why it matters Current answer
Can M03 advance to M03A analysis? M03A is supposed to analyze the completed 1M no-market-maker baseline or the final bounded gate. Not yet. Let the active 100k gate either complete or fail with preserved evidence.
Are stale-order lifecycle crashes fixed? These were hard blockers at batches 697 and 703. Promising but not proven to 100k. Batch 300 has expirations and no stale-order crash.
Is the receipt quarantine acceptable? It points at a local-chain observation versus domain-effect reconciliation gap. Unknown. It should become focused research if it persists or repeats.
Can the system sustain 100k and later 1M? Planner latency, live-order accumulation, and mirror lag are growing before the 100k target. Still under test. The current evidence says correctness is ahead of observability/performance.
Do we have liquidation or ADL replay coverage? The goal requires liquidations and potential ADL, not just passive order placement. No. The next replay family needs transformed prices and deliberately stressed margin states.
