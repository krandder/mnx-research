Engine egress: Quiver/TickSpread vs MNX

This version starts where it should have started: after the matching engine emits facts. The comparison is about how engine output becomes Kafka/Redpanda topics, fanout streams, projections, and finally consumable REST/WebSocket APIs.

Thesis

Quiver's book API was a MarketFeed projection of engine-level deltas on results. MNX currently has durable order/fill/batch facts and market/user notifications, but not an equivalent server-side per-price-level projection.

Engine output contract

results is the matching-engine fact stream

Boundary

The C matching engine opens a Kafka transaction, emits results_transaction_begin, writes engine facts, then emits results_transaction_commit. Consumers can treat the events between begin/commit as one engine auction/batch unit.

Identity

Results carry orders_offset and matching_engine_session_id. The fanout later adds results_offset, results_partition, source, fanout_session_id, and transaction begin offsets for replay safety. 

results transaction
  results_transaction_begin
    book_add / book_remove / book_remove_all        -> public orderbook deltas
    trade                                           -> public trades
    maker_order / delete_order / acknowledge_order  -> user/order lifecycle
    maker_trade / taker_trade / funding_payment     -> user/risk lifecycle
    execution_band / price_band / funding           -> market metadata
    auction_commit                                  -> auction boundary
  results_transaction_commit
book_add book_remove trade auction_commit maker_order delete_order maker_trade taker_trade funding

What fanout actually does

Fanout is a bridge, not the book service

Process Consumes Writes Important detail
risk_fanout.py risk_results orders, user_data Sends approved engine commands such as create_order, cancel_order, margin_update_order to orders; loops rejects, balances, user trades, and risk acks back to user_data.
fanout.py in matching-engine mode results user_data, orders Forwards user/order lifecycle events to user_data and turns auction_commit into auction_looped on orders. It ignores book_add, book_remove, book_remove_all, raw trade, bands, and funding for this path.
MarketFeed results Phoenix PubSub, stored slices This is the orderbook projection. It consumes engine book_add/book_remove deltas directly from results and broadcasts market updates.

Correction

The correct abstraction is not "the gateway asks the engine for a book." The engine emits durable book deltas on results; MarketFeed projects them into snapshots and updates; the fanout path preserves user/risk/order-loop state separately.

How API-consumable views are made

Quiver projections after engine egress

MarketFeed

Orderbook and market updates

MarketFeed.Consumer uses the transaction-aware AuctionManual consumer on results. It filters by market, waits for results_transaction_commit, applies OrderBook.update, creates a message via OrderBook.to_message, then broadcasts on market_feed:<symbol>:<precision>.

History

Candles and trade history

History consumes results directly. It samples auction_commit and forwards trade events into interval workers. This is a parallel projection, not a client-side reconstruction from open orders.

UserFeed

User orders, balances, positions

UserFeed consumes user_data, builds mutations for open orders, balances, open positions, trigger orders, and broadcasts user-specific socket events on user_data:<user_id>:<scope>.

Consumer API Source stream Semantics
Market data channel results -> MarketFeed On join: partial snapshot. Then update messages with level deltas/trades/status.
User data channel user_data -> UserFeed On join: balances, positions, orders. Then per-user lifecycle messages.
History/candles results -> History Trade and auction facts materialized independently of MarketFeed.

MNX details that matter

The closest MNX analogue is hidden in live state

Matchable quantity

MarketBook.getMatchableLiveOrders subtracts sealedQuantityByOrderHash and already filled quantity from live orders. This is the right semantic hook for a "what can match now" book.

Read model semantics

The materializer records batch_sealed lifecycle/fill rows at seal time, but order rows are updated by batch_settled. A REST book derived only from read_orders is not automatically the same thing as the live matchable book during a sealed-but-unsettled interval.

orders/by-market/:market_id Raw materialized order rows from read_orders, filtered by status/side/order type. The frontend requests open limit orders with a fixed high limit and aggregates by price locally. That is a UI reconstruction, not a server-side book projection contract.
MarketBook memory Live orders, filled quantities, sealed quantities, pending cancels, terminal markers. This has the right state for matchable book levels, but there is no public projection/API that exposes it as level snapshots and deltas.

Abstraction mismatch

The real gap is a missing MarketFeed-equivalent projection

Quiver

Engine emits book deltas

The matching engine itself emits public level deltas: book_add, book_remove, book_remove_all with market, side, price, precision, and visible delta. MarketFeed applies those events and serves snapshots/updates.

MNX

Engine emits order/fill facts

The FBA engine returns fills/order updates; HTV1 persists batch_sealed and batch_settled. Current consumers expose order rows and batch/user notifications, not an authoritative per-price-level stream.

Design implication

If MNX wants a Quiver-like orderbook API, the correct abstraction is not "make the frontend query more rows." It is a server-side projection that consumes the durable HTV1 fact stream and emits level snapshots/deltas with explicit sequence/read-after semantics.

Recommended MNX architecture

Build MarketFeed semantics over HTV1 facts

Component Contract Critical semantic choice
OrderBookLevels projection Maintains per-market, per-side, per-price aggregate quantities. Stores last journal cursor and publishes monotonic sequence. Decide mode=matchable vs mode=settled. The Quiver-like trading view should usually be matchable.
Apply order facts order_accepted adds visible quantity; order_cancelled/order_expired remove any remaining visible quantity. Keep conditional/pending-trigger orders out until they become accepted live orders.
Apply batch facts For matchable, decrement levels at batch_sealed by each order update's sealed quantity; re-add on retryable/unsent abort; clear in-flight state on batch_settled. Do not decrement again at settlement for the matchable mode. For settled, defer the decrement to batch_settled.
REST API GET /v1/markets/:id/orderbook?depth=&bucket=&mode=matchable returns bids/asks, sequence, read-after, and projection cursor. The response should be levels, not raw orders. Raw orders remain a separate diagnostic/order-management endpoint.
WebSocket API Emit ORDER_BOOK_SNAPSHOT and ORDER_BOOK_DELTA on a dedicated topic such as market:<id>:orderbook-levels. Keep market:<id>:orderbook compatibility as a refresh notification topic, or rename it after migration. 
MNX MarketFeed-equivalent projection
  trading-main journal
    order_accepted                  -> level += quantity
    order_cancelled / order_expired -> level -= remaining visible quantity
    batch_sealed                    -> level -= sealed_quantity   (matchable mode)
    batch_aborted_*                 -> level += restored sealed quantity
    batch_settled                   -> clear in-flight; publish trade metadata

  public output
    REST snapshot:  bids[], asks[], seq, read_after, mode, bucket
    WS deltas:      side, price, delta_quantity, seq, read_after
