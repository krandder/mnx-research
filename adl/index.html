ADL replacement-capacity invariants

These inequalities answer one narrow but important question: if an oracle update creates bankrupt exposure, can worst-case replacement traders on the other side still be valid ADL takers before the next oracle update?

Definitions Invariants Derivation Numbers What this does not prove

Short Version

If the exchange allows normal trading after an oracle update but before ADL finishes, old profitable opposite-side accounts can exit and be replaced by new max-leverage opposite-side accounts at the worst allowed trade price.

The two formulas below are a configuration sanity check. They ask whether those replacement accounts would still have a bankruptcy price on the safe side of the bankrupt maker's ADL execution price.

Upward oracle move, bankrupt shorts need long takers
(1 + J) * (1 + MTB_up) * (1 - IMR) <= 1
Downward oracle move, bankrupt longs need short takers
(1 - J) * (1 - MTB_down) * (1 + IMR) >= 1

What they buy

If they hold, the runtime does not need ADL to beat the first post-update normal trade. It only needs forced settlement to run before the next oracle update, assuming enough opposite-side quantity exists.

Definitions

J

The maximum oracle price jump per accepted oracle update.

Example: J = 2.5% means P_next can be at most 2.5% away from P.

MTB

The max trade bound: the allowed distance between oracle price and fill or clearing price.

MTB_up for upper-band buys; MTB_down for lower-band sells.

IMR

The initial margin requirement. For max-leverage positions, margin per unit is approximately IMR * entry.

IMR = 5% means max leverage is 20x.

The Invariants

Upward move

Price moves up. Existing shorts can become bankrupt. ADL closes those shorts against longs at the short's bankruptcy price.

worst long entry

Worst-case replacement long entry:

E_long = P * (1 + J) * (1 + MTB_up)

Max-leverage long bankruptcy price:

B_long = E_long * (1 - IMR)

To be usable as ADL taker for a short bankrupt around old P, this must be no higher than P.

Downward move

Price moves down. Existing longs can become bankrupt. ADL closes those longs against shorts at the long's bankruptcy price.

old P

E_short = P * (1 - J) * (1 - MTB_down)

B_short = E_short * (1 + IMR)

To be usable as ADL taker for a long bankrupt around old P, this must be no lower than P.

Derivation

Upward formula

1

Old oracle is P. New oracle is P * (1 + J).

2

Worst replacement long enters at P * (1 + J) * (1 + MTB_up).

3

A max-leverage long has bankruptcy price entry * (1 - IMR).

4

For ADL at the old short bankruptcy price, require B_long <= P.

P * (1 + J) * (1 + MTB_up) * (1 - IMR) <= P
(1 + J) * (1 + MTB_up) * (1 - IMR) <= 1

Downward formula

1

Old oracle is P. New oracle is P * (1 - J).

2

Worst replacement short enters at P * (1 - J) * (1 - MTB_down).

3

A max-leverage short has bankruptcy price entry * (1 + IMR).

4

For ADL at the old long bankruptcy price, require B_short >= P.

P * (1 - J) * (1 - MTB_down) * (1 + IMR) >= P
(1 - J) * (1 - MTB_down) * (1 + IMR) >= 1

Why 5% IMR plus 5% MTB is fragile

With IMR = 5% and MTB = 5%, the downward replacement condition already fails even when J = 0:

0.95 * 1.05 = 0.9975 < 1

That means a worst-edge max-leverage short can have a bankruptcy price below the old long bankruptcy price, so it may not be a valid ADL taker.

What This Proves And Does Not Prove

It proves, under the assumptions

  • Worst-case replacement longs remain eligible after an upward oracle jump.
  • Worst-case replacement shorts remain eligible after a downward oracle jump.
  • The runtime does not need to win a 200 ms race against the first post-update batch.
  • Forced settlement can be scheduled inside the guaranteed oracle-update interval.

It does not prove

  • Enough opposite-side quantity exists.
  • Backstop inheritance will succeed.
  • Book liquidation will succeed.
  • Binary 1x or other special cases are ADL-clearable.
  • Withdrawals or margin removal cannot reduce aggregate capacity unless separately gated.

Operational invariant that follows

After oracle update n, the runtime may allow normal trading during the interval only if the configured J, MTB, and IMR satisfy the replacement-capacity inequalities. Before accepting oracle update n + 1, all forced-settlement obligations created by update n must be cleared or the market must be in a protective mode.

Practical Recommendation For 5% MTB Markets

If MTB stays 5%

Raise the initial margin requirement to 8% and set the oracle max update to 2.5%. The downward side remains the tight side, but still passes:

(1 - 0.025) * (1 - 0.05) * (1 + 0.08) = 1.00035

This keeps the existing 5% trade bound while making the replacement-capacity invariant true for a 2.5% oracle step.

What changes economically

IMR = 8% means max theoretical leverage becomes 12.5x. Existing configs can still choose a lower default leverage such as 10x. If the product requires true 20x with 5% MTB, this invariant cannot be satisfied by lowering oracle jump alone.
