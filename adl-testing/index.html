The honest summary: the ADL/liquidation work is strong targeted regression coverage, not a completed production-scale liquidation campaign. Contract tests prove ADL behavior with real Solidity on local Hardhat. Runtime tests prove ADL planning, receipt rejection, journaling, and cancel-only state in synthetic exchange state. Local-chain replay proves order placement, relayers, settlement observation, and materializer plumbing at scale, but those scale runs were overfunded baseline runs and intentionally produced no liquidations or ADL.

We do not yet have a clean passing local-chain proof that starts from public API order placement, shocks price/funding, triggers ADL, mines a successful ADL transaction, observes the receipt, and applies every local effect.

Trigger Mechanics

Contract ADL

Open Position, Move Oracle, Call ADL

The clearest ADL trigger mechanics are in the Solidity tests. They mint and deposit collateral, open opposing positions, move the dummy oracle price, then call the ADL trader through Perpetual.trade.

Examples: long ADL opens at price 100, moves oracle down to 70 or 89.1, and deleverages the bankrupt maker. Short ADL moves oracle up, for example 100 to 130. Funding tests also advance time and set funding so a taker that looked solvent becomes underwater inside the ADL call.

Runtime ADL

Seed Underwater State, Then Run Planner

The Bun runtime tests do not open real positions on-chain. They seed exchange state directly. A representative setup is maker size 10, margin 90, entry price 100, oracle price 89, plus an opposing taker.

The test then calls triggerAutomaticDeleveragingPass. The expected result is one planned ADL round, one submission, market cancel-only pinned, and a relayer submission captured by the stubbed relayer.

Local-Chain Trigger

M01B Funding Shortfall

One disposable local-chain run created an actual AAPL position through the public API: 2 @ 276.5, long at 10x against a short at 2x.

It advanced Anvil time by 3605s, crystallized a funding rate, then jumped another 1112388s so the funding debit overtook margin. Funding-adjusted equity became negative, the liquidation monitor found one underwater position with MR=-0.0300, book liquidation skipped it because equity was non-positive, and the ADL path was expected.

Not Yet Clean

Local-Chain ADL Did Not Finish Cleanly

The M01B local-chain trigger produced useful signal but not a clean ADL settlement proof. The notes record that ADL relayer broadcasts were accepted by the runtime relayer but reverted on receipt after the large time warp.

So the honest status is: contract ADL success is tested; runtime ADL planning and receipt handling are tested; a full local-chain ADL lifecycle from price/funding shock through successful mined ADL receipt remains open.

Evidence

Research Loop

The replay orchestrator keeps the audit trail. Each step has one Axx_STRESS_TEST.md file, one Axx_ANALYSIS.md file, and a run folder with the command, red result, fix commit, and green verification.

analysis red test fix full-suite verify

The harness pattern is deliberately narrow: seed a submitted ADL state, inject one adversarial decoded receipt through executionBackend.getReceipt, call observeSubmittedAdlReceipts, then assert that no position, balance, PnL, or adl_receipt_applied facts were journaled when the receipt should be rejected.

Liquidation-Specific Testing

Bug Found

A33: Liquidation Contamination

A33 tested whether an otherwise ADL-shaped successful receipt could carry decoded liquidation_trade_updates and still apply the ADL-only view. Before the fix, this cross-family contamination was not rejected early enough.

Fix commit a083ea79 rejects contaminated ADL liquidation receipts before any ADL receipt effects are journaled.

Scope Note

What "Liquidation" Means Here

This workstream did not try to exhaustively fuzz the standalone liquidation engine. The liquidation coverage here is targeted at the ADL receipt boundary: liquidation-family decoded events must not contaminate ADL receipt application, source-event keys, or accounting effects.

Related cross-family checks also cover regular batch-trade contamination, backstop inheritance sidecars, forbidden event names, and batch-trade-count contamination.
